## Часть 1. Проектирование архитектуры

### 1. Тип приложения
Разрабатываемая система представляет собой распределённое веб-приложение для электронной коммерции — **Маркетплейс**. Архитектура реализована в соответствии со стилем **микросервисов (Microservices)**, где каждый сервис отвечает за отдельную бизнес-функцию (управление товарами, обработка заказов, работа с пользователями и т.д.). Клиентское взаимодействие осуществляется через **REST API**, а также через асинхронные события для внутренних процессов.

### 2. Стратегия развёртывания
Выбрана стратегия **контейнеризации** с использованием **Docker** и оркестрации.
- **Контейнеризация** позволяет гибко масштабировать отдельные компоненты системы (например, сервис каталога товаров) независимо от остальных.
- **Оркестрация:** Использование **Kubernetes** для автоматического развертывания, масштабирования и управления отказоустойчивостью контейнеров.

### 3. Выбор технологии
- **Backend:** Java 24, Spring Boot 3.5.0.
- **Базы данных:**
  - **PostgreSQL:** Основная реляционная БД для пользователей, заказов, платежей.
  - **MongoDB:** Для каталога товаров, где структура данных может часто меняться (разные категории товаров имеют разные атрибуты).
  - **Redis:** Кэширование популярных товаров, корзины покупателя, хранение сессий.
- **Брокер сообщений:** Apache Kafka.
  - *Обоснование:* Необходим для обработки событий (события домена). Kafka обеспечивает надёжную доставку и хранение событий.
- **Аутентификация:** Spring Security.
  - *Обоснование:* Стандартная регистрация, централизованное управление пользователями, использование JWT-токенов.

### 4. Показатели качества
- **Производительность:** Время загрузки каталога товаров < 500 мс. Время оформления заказа < 2 секунд.
- **Масштабируемость:** Возможность выдерживать пиковые нагрузки 10 000+ одновременных пользователей.
- **Доступность:** Система должна оставаться работоспособной даже при отказе отдельных сервисов (например, при падении сервиса рекомендаций пользователь всё ещё может оформить заказ).
- **Безопасность:** Защита персональных данных пользователей, безопасность платежных транзакций.

### 5. Сквозная функциональность
- **Логирование:** Централизованный сбор логов. Каждый запрос имеет уникальный traceId для отслеживания пути через все микросервисы.
- **Обработка ошибок:** Глобальные обработчики исключений (`@RestControllerAdvice`), возвращающие клиенту понятные сообщения об ошибках (например, "Товар закончился на складе").
- **Аутентификация и авторизация:** JWT-токены с ограниченным временем жизни. Проверка прав доступа через Spring Security. API Gateway проверяет валидность токена на входе.

## Часть 2. Анализ архитектуры
После завершения первого спринта в рамках разработки маркетплейса были реализованы два микросервиса: **Auth Service** и **User Service**.
- **Auth Service** отвечает за регистрацию пользователей, вход в систему, генерацию и валидацию JWT-токенов, а также хеширование паролей с использованием BCrypt.
- **User Service** предоставляет CRUD-операции для управления профилями пользователей, включая получение данных по ID и email, обновление и удаление профилей.

## Часть 3. Сравнение и рефакторинг

### 1. Сравнение архитектур «As is» и «To be»
Для анализа мы берем два уровня абстракции.
- **Целевая архитектура «To be»** из Части 1 представляет собой высокоуровневую диаграмму компонентов, показывающую идеальную структуру маркетплейса с полным набором сервисов (Auth, User, Payment, Order), общими базами данных (PostgreSQL, Redis) и брокером сообщений (Kafka).
- **Текущая реализация «As is»** из Части 2 — это детальные диаграммы классов двух реализованных сервисов (`auth-service`, `user-service`), демонстрирующие их внутреннее устройство.

При сравнении видно, что общий архитектурный стиль совпадает: оба представления ориентированы на микросервисную архитектуру. Также успешно реализовано разделение на Auth Service и User Service, что соответствует плану. Auth Service действительно занимается генерацией и валидацией JWT, а User Service предоставляет CRUD-операции для управления профилями.

### 2. Анализ отличий и их причины
- **Отсутствие Gateway.** Вместо того чтобы проверять токены в одном общем месте (шлюзе), сейчас каждый сервис делает это сам. В User Service есть специальный фильтр для проверки JWT. На старте проще добавить проверку в сам сервис.
- **Отсутствие Kafka.** Причина — стремление избежать излишней сложности на раннем этапе.

### 3. Пути улучшения архитектуры (Рефакторинг)
1.  **Внедрение API Gateway.** Вынести проверку JWT, логирование и маршрутизацию в отдельный шлюз. Это упростит сервисы и сделает их более сфокусированными.
2.  **Интеграция Kafka.** Настроить событийное взаимодействие: при регистрации пользователя Auth Service публикует событие, а User Service создает профиль, подписываясь на него. Это обеспечит слабую связность и консистентность данных.

## Вывод
В целом, различия между текущей реализацией и целевой архитектурой не большие. Основная структура правильная: сервисы разделены, каждый отвечает за свою задачу, Auth Service работает с токенами, User Service управляет профилями. Главные недочеты связаны с тем, что пока не хватает некоторых важных элементов. Также нужно четче определить, где какие данные хранятся. На следующих этапах разработки недостающие компоненты добавятся, границы сервисов уточнятся.
