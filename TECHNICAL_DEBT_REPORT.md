## 1. АНАЛИЗ ПО ПРИЗНАКАМ ТЕХНИЧЕСКОГО ДОЛГА

### 1.1 Непонятный / нечитабельный код

| Проблема | Пример | Критичность |
|----------|--------|-------------|
| **Смешение логики в API endpoints** | `backend/app/api/orders.py` — бизнес-логика создания заказа внутри route-функции | Средняя |

**Оценка**: Код в целом читаемый, но требует рефакторинга для выделения бизнес-логики в сервисный слой.

## 5. ВЫВОДЫ И РЕКОМЕНДАЦИИ

### 5.1 Необходимость устранения долга

| Аргумент | Обоснование |
|----------|-------------|
| **✅ Устранять необходимо** | Без тестов невозможно безопасное добавление фич |
| **✅ Критично для команды** | При масштабировании команды долг станет блокирующим |
| **⚠️ Частично для учебного проекта** | Для демонстрации функционала текущее состояние приемлемо |

### 5.2 Корректировка плана разработки

**Рекомендуемый подход**: **Параллельное выполнение**

```
Спринт 1-2:  Приоритет 1 (тесты + CI) + 1 фича (небольшая)
Спринт 3-4:  Приоритет 2 (рефакторинг) + 1 фича (средняя)
Спринт 5:    Приоритет 3 (документация) + интеграция Kafka
```

### 5.3 Риски без устранения долга

| Риск | Вероятность | Влияние |
|------|-------------|---------|
| Регрессии при изменениях | Высокая | Критическое |
| Невозможность масштабирования | Средняя | Высокое |
| Сложность онбординга новых разработчиков | Высокая | Среднее |
| Проблемы с БД при деплое | Высокая | Критическое |

## 7. ЗАКЛЮЧЕНИЕ

### Текущее состояние проекта

Для учебного/демонстрационного проекта — **удовлетворительное**. Функционал работает, архитектура микросервисная (хотя и упрощённая).

### Критические проблемы

1. **Полное отсутствие тестов** — невозможно гарантировать работоспособность при изменениях
2. **База данных в репозитории** — нарушение базовых принципов Git
3. **Нет CI/CD** — ручная интеграция и деплой
4. **Нет миграций БД** — создание таблиц при старте не подходит для production

### Рекомендация

> **Перед добавлением новых фич обязательно реализовать Приоритет 1** (тесты + CI + миграции). Это займёт 2-3 недели, но сэкономит время в будущем и обеспечит стабильность разработки.
