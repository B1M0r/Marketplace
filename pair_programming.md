<a name="intro"/>
# 1 Введение
В рамках лабораторной работы мы реализовали **первые микросервисы** (Auth Service + User Service) и начальную часть **фронтенда** онлайн-маркетплейса.  
Документ описывает процесс парного программирования, результаты работы и выводы.

**Технический стек:**
- **Backend:** Java 24 + Spring Boot 3.5.0 + Spring Security + JWT + BCrypt
- **Frontend:** React + Axios (JavaScript)
- **Базы данных:** PostgreSQL (пользователи), Redis (сессии)
- **Контейнеризация:** Docker + Docker Compose

<a name="development_stages"/>
## 2.2 Этапы разработки

| Итерация | Направление          | Действия |
|:---------|:---------------------|:---------|
| **1**    | Бэкенд → Фронтенд    | • Бэкенд-разработчик создал Auth Service с эндпоинтами `/auth/register` и `/auth/login`<br>• Бэкенд-разработчик реализовал генерацию и валидацию JWT-токенов<br>• Фронтенд-разработчик создал страницы `Login` и `Register`, подключил их к API |
| **2**    | Фронтенд → Бэкенд    | • Фронтенд-разработчик добавил компонент авторизации с сохранением токена (Context + localStorage)<br>• Бэкенд-разработчик создал User Service с CRUD-операциями для профилей пользователей |
| **3**    | Бэкенд → Фронтенд    | • Бэкенд-разработчик реализовал защиту эндпоинтов через JWT-фильтр и роли (USER/ADMIN)<br>• Фронтенд-разработчик создал `ProtectedRoute` и условное отображение интерфейса по роли |
| **4**    | Фронтенд → Бэкенд    | • Фронтенд-разработчик добавил форму редактирования профиля<br>• Бэкенд-разработчик доработал User Service и настроил Docker Compose для совместного запуска |

# 3 Результаты работы

<a name="implemented_features"/>
## 3.1 Реализованный функционал

| № | Функция                          | Требования |
|---|----------------------------------|------------|
| 1 | Регистрация пользователей        | Создание аккаунта с хэшированием пароля |

| 7 | Docker-окружение                 | Запуск всех сервисов одной командой |

<a name="positive_experience"/>
## 3.2 Положительные впечатления
1. **Обмен знаниями** — бэкенд-разработчик лучше разобрался в React (хуки, контекст), фронтенд-разработчик глубже понял Spring Security и JWT.
2. **Скорость разработки** — не нужно было долго согласовывать API, всё работало сразу после каждой итерации.
3. **Качество кода** — каждая строчка проходила двойную проверку в процессе написания.
4. **Меньше багов** — ошибки с токенами, CORS и ролями находились практически мгновенно.
5. **Интересный опыт** — работать в паре над fullstack-задачей оказалось намного увлекательнее.

<a name="problems"/>
# 4 Проблемы и их решения

| Проблема | Решение |
|:---------|:--------|
| Разные версии Java/Node и проблемы с портами | Создали `RUN_LOCAL.md` и унифицировали всё через Docker Compose |
| CORS-ошибки при подключении фронтенда | Добавили правильную CORS-конфигурацию на бэкенде |
| Задачи разного объёма в стиле пинг-понг | Разбивали работу на небольшие примерно равные итерации |
| Хранение и автоматическое обновление токена | Совместно реализовали AuthContext |
| Забывание менять роли | Поставили таймер на 25–30 минут |

<a name="conclusions"/>
# 6 Выводы
Парное программирование оказалось **эффективным инструментом**, особенно для задач, требующих тесной интеграции между бэкендом и фронтендом. Стиль **"Пинг-понг"** идеально подошёл для старта нашего маркетплейса.

## Плюсы и минусы

| ✅ Плюсы                              | ❌ Минусы |
|---------------------------------------|----------|
| Значительно меньше ошибок             | Требует синхронизации времени |
| Быстрый обмен знаниями между стеками  | Устаёшь быстрее при долгой работе |
| Высокое качество кода и архитектуры   | Нужно привыкнуть к постоянному общению |
| Быстрая интеграция бэкенда и фронтенда| — |

В целом опыт получился очень полезным. Мы **планируем активно использовать парное программирование** при дальнейшей разработке проекта (API Gateway, Product Service, Order Service и т.д.).
