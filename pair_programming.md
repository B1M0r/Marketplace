<a name="intro"/>
# 1 Введение
В рамках лабораторной работы мы реализовали **первые микросервисы** (Auth Service + User Service) и начальную часть **фронтенда** онлайн-маркетплейса.  
Документ описывает процесс парного программирования, результаты работы и выводы.

**Технический стек:**
- **Backend:** Java 24 + Spring Boot 3.5.0 + Spring Security + JWT + BCrypt
- **Frontend:** JavaScript + React (или Next.js — в зависимости от текущей реализации)
- **Базы данных:** PostgreSQL (пользователи), Redis (сессии)
- **Контейнеризация:** Docker + Docker Compose

<a name="development_stages"/>
## 2.2 Этапы разработки

| Итерация | Направление          | Действия |
|:---------|:---------------------|:---------|
| **1**    | Бэкенд → Фронтенд    | • Бэкенд-разработчик создал Auth Service с эндпоинтами `/auth/register` и `/auth/login`<br>• Бэкенд-разработчик реализовал генерацию и валидацию JWT-токенов<br>• Фронтенд-разработчик создал страницы `Login` и `Register`, подключил их к API через Axios/Fetch |
| **2**    | Фронтенд → Бэкенд    | • Фронтенд-разработчик добавил компонент авторизации с сохранением токена (localStorage / Context)<br>• Бэкенд-разработчик добавил User Service с CRUD-операциями для профилей пользователей |
| **3**    | Бэкенд → Фронтенд    | • Бэкенд-разработчик реализовал защиту эндпоинтов через JWT-фильтр и базовые роли (USER/ADMIN)<br>• Фронтенд-разработчик создал ProtectedRoute и отображение разного интерфейса в зависимости от роли |
| **4**    | Фронтенд → Бэкенд    | • Фронтенд-разработчик добавил форму редактирования профиля пользователя<br>• Бэкенд-разработчик доработал User Service (обновление и удаление профиля) и настроил Docker Compose для совместного запуска |

<a name="results"/>
# 3 Результаты работы

<a name="implemented_features"/>
## 3.1 Реализованный функционал

| № | Функция                          | Требования |
|---|----------------------------------|------------|
| 1 | Регистрация пользователей        | Создание аккаунта с хэшированием пароля (BCrypt) |
| 2 | Авторизация                      | Вход с выдачей JWT-токена |
| 3 | Валидация JWT                    | Проверка токена на бэкенде и фронтенде |
| 4 | CRUD профилей пользователей      | Получение, обновление, удаление профиля |
| 5 | Базовое разграничение прав       | Роли USER/ADMIN |
| 6 | Защищённые маршруты на фронтенде| ProtectedRoute / AuthContext |
| 7 | Docker-окружение                 | Совместный запуск бэкенда и фронтенда |

<a name="positive_experience"/>
## 3.2 Положительные впечатления
1. **Обмен знаниями** — бэкенд-разработчик глубже разобрался в React (хуки, контекст, маршрутизация), фронтенд-разработчик лучше понял Spring Security и JWT.
2. **Скорость разработки** — не тратили время на долгие согласования API, всё сразу работало после каждой итерации.
3. **Качество кода** — двойная проверка «на лету» сильно снизила количество ошибок.
4. **Меньше багов при интеграции** — благодаря постоянному переключению ролей проблемы с токенами и CORS находились сразу.
5. **Интересный и увлекательный опыт** — работать в паре над полным стеком гораздо веселее, чем по отдельности.

<a name="problems"/>
# 4 Проблемы и их решения

| Проблема | Решение |
|:---------|:--------|
| Сложности с настройкой окружения (разные версии Java/Node, порты) | Создали подробный `RUN_LOCAL.md` и унифицировали версии через Docker Compose |
| CORS-ошибки при подключении фронтенда к бэкенду | Бэкенд-разработчик добавил CORS-конфигурацию, фронтенд — правильные заголовки |
| Задачи разного объёма в стиле "пинг-понг" | Разбивали большие задачи на мелкие итерации примерно равной длительности |
| Хранение и обновление JWT-токена на фронтенде | Вместе реализовали AuthContext + автоматическое обновление |
| "Залипание" и забывание менять роли | Поставили таймер на 25–30 минут для смены ведущего |

<a name="conclusions"/>
# 6 Выводы
Парное программирование оказалось **очень эффективным инструментом**, особенно для задач, требующих тесной интеграции между бэкендом и фронтендом. Стиль **"Пинг-понг"** идеально подошёл для нашей работы, так как позволил синхронно развивать оба слоя приложения.

## Плюсы и минусы

| ✅ Плюсы                              | ❌ Минусы |
|---------------------------------------|----------|
| Значительно меньше ошибок и багов     | Требует синхронизации графиков работы |
| Быстрый обмен знаниями между стеками  | Устаёшь быстрее, чем при соло-работе |
| Высокое качество кода и архитектуры   | Нужно привыкнуть к постоянному общению |
| Быстрая и качественная интеграция     | — |

В целом опыт получился очень полезным и позитивным. Мы **планируем активно использовать парное программирование** при дальнейшей разработке маркетплейса (API Gateway, Product Service, Order Service, оформление заказа и т.д.).
